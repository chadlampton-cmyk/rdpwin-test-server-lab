Major findings
- Pure RemoteApp is not viable for RDPWin in this lab.
- Full AVD desktop session works and is the active user-path model.
- Explorer must remain running. Shell replacement, Explorer kill, and aggressive Start/taskbar lockdown destabilized RDPWin.
- The Zen license issue on DBTEST01 was real, but it was not the root cause of the RemoteApp crash.
- The current HKLM Run launcher is unreliable across Entra/AVD users.

Zen licensing
- Actian CLI showed the installed Zen 16 Cloud Server temporary license had expired on 2026-04-15.
- License was reactivated later in the session.
- Btrieve Error 161 cleared after reactivation.
- RemoteApp still crashed after the license fix, confirming license state was not the main RemoteApp blocker.

User-path findings
- Chad’s Entra session hit the launcher and logged RDPWin start/exit behavior.
- Felix is not a local admin on RDPDISC01.
- Felix did not hit the launcher at all, which points to the current trigger mechanism rather than admin bypass.
- Bastion remains the admin path only, not the target customer/user path.

Current target architecture
- AVD desktop stays as the user path.
- Separate admin and non-admin identities.
- Replace HKLM Run with a Scheduled Task at user logon.
- Keep Explorer alive.
- Launch RDPWin deterministically at logon for non-admin users.
- Log off the session fully when RDPWin closes.
- Use Entra security groups as the source of truth for database routing.
- Add a local broker on RDPDISC01 to choose the correct DB/config for the user at launch.
- Do not let users browse or select DB paths directly.

What was ruled out
- RemoteApp as the final access model.
- Bastion as the user-path test model.
- Shell replacement as the immediate solution.
- Unsupported Start-menu hacks as a reliable control.
